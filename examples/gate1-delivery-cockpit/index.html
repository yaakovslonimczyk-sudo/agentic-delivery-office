Unresolved Blocking Finding
security-reviewer Race condition on concurrent stock updates — SEVERITY HIGH

When two concurrent requests decrement the same SKU quantity, the read-modify-write is not atomic. Under load (>40 rps on a single SKU) inventory can go negative, enabling oversell.

Evidence: src/inventory/stock_service.ts:L214–L228 — no optimistic lock or DB-level serialization around updateQty()

Refuter verdict: CONFIRMED — not a false positive Refuter: architectural-reviewer agent, 2026-06-11 20:12 UTC

Fix applied: added SELECT FOR UPDATE advisory lock around updateQty(). Re-reviewer finding: lock is acquired after the initial SELECT, so the TOCTOU window remains. Concurrent reads still see stale value before lock acquisition.
ROUND 2 · 2026-06-11 21:15 UTC ✕ re-review: still blocked
Fix applied: moved to single atomic UPDATE qty = qty - $delta WHERE qty >= $delta RETURNING qty SQL pattern. Re-reviewer finding: fix is correct at DB level but stock_service.ts:L189 caches qty in Redis with a 30-second TTL — cache is not invalidated in the hot path, so the application layer still serves stale inventory to clients even after the DB fix.
